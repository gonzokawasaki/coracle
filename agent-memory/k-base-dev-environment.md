For [[k-base-project]] (AMAdocs). **Phase 0 foundation is PROVEN working end-to-end (2026-06-12)**: dropped a markdown file in → parsed → local ONNX embeddings → LanceDB → retrieved → answered by local LLM with a correct doc-only fact ("27.4°C, Dr. Penelope Quirkstone") and cited the source. Full local/offline RAG works.

**Layout (all under `/mnt/space/k-base`):**
- `anythingllm-upstream/` — the forked AnythingLLM monorepo (server/collector/frontend).
- `tooling/ollama/bin/ollama` — userspace Ollama 0.30.7 (no sudo). Models in `tooling/ollama-models/`.
- `tooling/start-stack.sh` — launches server(3001)+collector(8888)+frontend(3000); logs in `tooling/logs/`.
- `tooling/test-docs/` — sample docs for testing.

**Run it:** Node 18.18.0 required (via nvm; native modules don't build on Node 26). Start Ollama: `OLLAMA_MODELS=/mnt/space/k-base/tooling/ollama-models OLLAMA_HOST=127.0.0.1:11434 nohup tooling/ollama/bin/ollama serve &` then pull/use `qwen2.5:3b`. Then `bash tooling/start-stack.sh`. UI at http://localhost:3000. API is unauthenticated in dev single-user mode.

**Default model:** swapped Qwen2.5-3B (research/non-commercial license) → **phi3.5** (Phi-3.5-mini, MIT, ~2.2GB) for clean OSS licensing. Both pulled in Ollama. CAVEAT: Phi-3.5 answers correctly but is noticeably MORE VERBOSE/rambly than Qwen2.5-3B — needs a "be concise" system prompt for good UX. License rundown done: app=MIT, all deps MIT/Apache; only the model license matters → use Apache/MIT models → no EULA checkbox needed, just bundle THIRD_PARTY_LICENSES + About/Licenses screen. Planned: hardware-tiered model auto-select (1.5B weak / 3.8B default / 7B Apache strong).

**Fork change (doc viewer):** added GET `/workspace/:slug/doc-view?path=<docpath>` to `server/endpoints/workspaces.js` (uses `fileData()`, path-traversal safe) → returns {title,pageContent,wordCount}. UI: sidebar docs are clickable → open Obsidian-style split reading pane (`#viewer` beside `#chat`) showing the extracted text the AI reads (originals NOT retained, so no PDF-layout render yet). Editing server code requires an Electron restart (server runs as plain `node`, not nodemon).

**Restart gotcha:** DON'T use `pkill -f`/`pgrep -f` with the electron path — it matches this agent's own shell and kills it (exit 144/1). Use explicit PIDs from `ss -ltnp`, or `setsid nohup ... electron .` to launch. Orphaned electron renderer procs hold no ports; harmless.

**Reveal-in-explorer + resizable panels (added 2026-06-12):** `amadocs-desktop/preload.js` exposes `window.amadocs` via contextBridge: `revealInFolder(path)`→ipc→`shell.showItemInFolder`, and `pathForFile(file)`→`webUtils.getPathForFile` (Electron 31; sandbox:false + preload set in main.js). KEY CONSTRAINT: the engine does NOT retain where an uploaded file came from (collector extracts text then `trashFile`s the original; stored `url` is the temp hotdir path). So the UI captures the original absolute path **client-side at drop/choose time** (`resolveFilePath`) into a `localStorage` map `amadocs.filePaths` keyed by **filename**, used by both the sidebar right-click "Open in system explorer" menu and clickable source-citation links in answers. Limitation: only works for files added after this change, matches by filename (breaks if a converter sets a different title). A robust fix later = persist originalPath in document metadata server-side. Drop = one-way derived COPY, never a link. Sidebar + reading-pane/chat split are mouse-resizable (drag `.gutter`/`#gutterSplit`, widths persisted in localStorage `amadocs.sidebarW`/`amadocs.viewerW`).

**Naming + delete + copy-clarity (added 2026-06-12):** The doc collection is named **"Library"** in the UI (decided over "vault" — clearest for non-tech users). A `.libnote` in the sidebar + dropzone hint state the originals stay put and AMAdocs keeps its own private copy. Delete: per-doc hover 🗑️ + right-click "Remove from Library", and a "Clear all" button in the Library header — all call `POST /workspace/:slug/update-embeddings` with `{deletes:[docpaths]}` (docpath stored on each row's `li.dataset.docpath`), with `confirm()` dialogs and path-map cleanup. KNOWN LIMITATION / FOLLOW-UP: the doc viewer is **text-only** — no images/graphics/layout (collector extracts only text then trashes the original). The vnote now says so. To show graphics later = retain the original binary + render (PDF.js etc.); this is the same root cause as reveal-in-explorer needing client-side path capture.

**Pretty viewer BUILT + verified E2E (2026-06-12)** — supersedes the text-only limitation above. (1) Collector now RETAINS originals: `collector/utils/files` adds `originalsFolder` + `stashOriginal/commitOriginal/discardOriginal`; `processSingleFile` stashes the source before the converter trashes it, commits as `server/storage/originals/<docId><ext>`. (2) Server: `GET /workspace/:slug/doc-original?path=` streams it with content-type (switch on blob type client-side); `update-embeddings` deletes the matching original on doc removal. (3) UI viewer renders by type — PDF via bundled **PDF.js v3 UMD** (lazy per-page canvas render via IntersectionObserver), images as `<img>`, docx via **mammoth**, xlsx via **SheetJS**; plus a "Original ↔ Text the AI reads" toggle (text view = existing doc-view). Libs vendored offline in `tooling/amadocs-ui/vendor/` (pdf.min.js, pdf.worker.min.js, mammoth.browser.min.js, xlsx.full.min.js) → copied to `amadocs-desktop/ui/vendor/`. Verified by API: upload→original retained→doc-original byte-identical→delete cleans up. (4) **Per-page tracking**: `asPDF` already loads with `splitPages:true` (page nums in `doc.metadata.loc.pageNumber`) but joined+discarded them — now also writes `pages:[{page,start,end}]` (char ranges into pageContent), surfaced via `doc-view`. **Config:** `server/.env.development` set to LLM_PROVIDER=ollama, EMBEDDING_ENGINE=native (Xenova/all-MiniLM-L6-v2), VECTOR_DB=lancedb, DISABLE_TELEMETRY=true.

**Perf on GTX 1650 Ti (3.5GB VRAM):** qwen2.5:3b runs 100% on GPU. Cold start (first query/model load into VRAM) ~70s; warm queries ~1.2s. Mitigation: keep model resident (OLLAMA_KEEP_ALIVE) + "warming up" splash on first launch.

**Desktop wrapper (the "build"):** `amadocs-desktop/` — Electron 31 app. `main.js` spawns Ollama + server + collector as **Node-18 child processes** (NOT in-process: engine native modules are ABI-built for Node 18, not Electron's Node), shows `loading.html` splash, health-checks 3001, then loads `ui/index.html` (copy of the AMAdocs UI). Reuses an already-running dev stack if present. Cold boot from killed ports → engine up in ~8s, confirmed 2026-06-12, renders in Electron's own Chromium on Wayland w/ GPU. Run: `cd amadocs-desktop && WAYLAND_DISPLAY=wayland-1 DISPLAY=:1 ./node_modules/.bin/electron .` (node18 via nvm). Paths in main.js are dev-layout (`../anythingllm-upstream`, `../tooling/ollama`) — must be replaced with bundled paths for packaging (task: AppImage). UI CONFIRMED "exactly right" by user 2026-06-12 (in Electron window) — direction locked; only small polish left (collections switcher, About/Licenses screen). Build the installer around `ui/index.html`, not a rework. UI source of truth: `tooling/amadocs-ui/index.html` (copied to `amadocs-desktop/ui/`).

**Gotcha:** new workspaces default to chatMode "automatic" which auto-routes tool-calling models into agent mode; set chatMode "query" (docs-only, anti-hallucination) or "chat" for normal RAG.
